# Feature Specification: Full Moltis-Native Codex Update Skill

**Feature Branch**: `023-full-moltis-codex-update-skill`  
**Created**: 2026-03-14  
**Status**: Draft  
**Input**: Follow-up after `012-codex-upstream-watcher`, `021-moltis-native-codex-update-advisory`, and `moltinger-ewde`, based on the corrected product goal that the entire Codex update capability must live inside Moltis/Moltinger as its own skill/agent rather than remain a repo-side runtime with Moltis-only delivery.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Moltis Answers Codex Update Questions Directly (Priority: P1)

A user asks Moltis in plain language to check Codex CLI updates, and Moltis itself performs the upstream check, reasons about the result, and answers in Russian without depending on repo-side watcher scripts as the canonical runtime.

**Why this priority**: This is the core correction. The skill must belong to Moltis, not to a repo-side shell layer.

**Independent Test**: Ask Moltis a natural-language request such as "Проверь обновления Codex CLI" and confirm that the result is generated by the Moltis-native skill path.

**Acceptance Scenarios**:

1. **Given** the user asks Moltis to check Codex CLI updates, **When** the skill runs, **Then** Moltis fetches the official source, classifies the result, and replies in Russian.
2. **Given** the upstream source is unchanged since the last snapshot, **When** the user asks again, **Then** Moltis still answers correctly but does not create a false "new update" event.
3. **Given** the upstream source is unavailable or malformed, **When** the skill runs, **Then** Moltis returns an honest `нужно проверить` style result rather than a guessed verdict.

---

### User Story 2 - Moltis Monitors Upstream Codex Releases On Its Own Schedule (Priority: P1)

Moltis tracks new upstream Codex CLI releases without needing the repo-side watcher script to be the authoritative scheduler/runtime owner.

**Why this priority**: A full transfer into Moltis is incomplete unless polling, fingerprint state, and alert decisions also move into Moltis.

**Independent Test**: Trigger the Moltis-native scheduled path against a fixture or fresh upstream source and confirm Moltis creates at most one alert for one new fingerprint.

**Acceptance Scenarios**:

1. **Given** Moltis runs its Codex update scheduler, **When** a fresh upstream fingerprint appears, **Then** Moltis records it and produces one advisory event.
2. **Given** the same fingerprint is seen again, **When** the scheduler runs, **Then** Moltis suppresses duplicate delivery.
3. **Given** the scheduler is disabled or degraded, **When** operators inspect the state, **Then** they can see why no delivery was attempted.

---

### User Story 3 - Moltis Adapts Recommendations To A Project Profile (Priority: P2)

Moltis can optionally tailor Codex update recommendations to a project by reading a stable profile/manifest rather than relying on repo-side runtime logic as the canonical advisory engine.

**Why this priority**: The user wants one Moltis-owned skill. Project context should be an optional input, not the place where the skill lives.

**Independent Test**: Provide one project profile and confirm Moltis produces project-specific recommendations; remove the profile and confirm Moltis still produces a generic upstream advisory.

**Acceptance Scenarios**:

1. **Given** a valid project profile is available, **When** Moltis evaluates a new Codex release, **Then** the follow-up includes project-specific recommendations.
2. **Given** no project profile is available, **When** Moltis evaluates a new release, **Then** it still sends a generic but useful upstream advisory.
3. **Given** the project profile is malformed, **When** Moltis evaluates a new release, **Then** it degrades safely and records the profile error without blocking the upstream alert.

---

### User Story 4 - Operators Can Audit And Roll Back The Skill Safely (Priority: P2)

Operators can prove that the skill, scheduler, state, alerting, and optional project-profile path are all Moltis-owned, and can safely roll back to one-way alerts or manual checks if needed.

**Why this priority**: The current repo contains several intermediate layers; operators need one clear ownership model and a safe rollback.

**Independent Test**: Inspect one machine-readable interaction/state record and confirm it was produced by the Moltis-native skill path, not the legacy repo-side watcher path.

**Acceptance Scenarios**:

1. **Given** a complete advisory run, **When** an operator inspects the audit/state record, **Then** they can see upstream fingerprint, skill decision, delivery status, and profile applicability source.
2. **Given** the interactive path is disabled, **When** Moltis sends an alert, **Then** it degrades to one-way mode without advertising a broken flow.
3. **Given** operators disable the new skill path, **When** they roll back, **Then** the system returns to a documented fallback without reviving retired `/codex_*` UX.

### Edge Cases

- What happens when the official Codex changelog format changes?
- What happens when advisory issue signals disagree with the official changelog?
- What happens when several project profiles request different applicability rules for the same release?
- What happens when one project profile exists but the upstream release is not actionable?
- What happens when Telegram delivery succeeds but profile-specific recommendation rendering fails?
- What happens when Moltis is asked on demand while a scheduled run is already processing the same fingerprint?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Moltis MUST become the canonical runtime owner of Codex update detection, state, alerting, and user-facing query handling.
- **FR-002**: Repo-side watcher/runtime scripts MUST NOT remain the canonical production path for Codex update checking once this feature lands.
- **FR-003**: Moltis MUST support an on-demand user-facing skill/agent entrypoint for plain-language Codex update requests.
- **FR-004**: Moltis MUST support a scheduled upstream polling mode owned by Moltis itself.
- **FR-005**: Moltis MUST track a stable upstream fingerprint/state to suppress duplicate alerts.
- **FR-006**: Moltis MUST prefer the official Codex CLI source as primary truth.
- **FR-007**: Optional advisory issue signals MAY enrich context but MUST NOT replace the primary official source.
- **FR-008**: Human-facing summaries and alerts for this feature MUST be in Russian.
- **FR-009**: Moltis MUST support Telegram delivery for fresh actionable Codex update events.
- **FR-010**: Moltis MUST support a safe one-way degraded mode when interactive follow-up is unavailable.
- **FR-011**: Moltis SHOULD support an optional project profile/manifest contract for project-specific applicability.
- **FR-012**: If no project profile exists, Moltis MUST still provide a useful generic upstream advisory.
- **FR-013**: If a project profile is malformed, Moltis MUST record the error and degrade safely without blocking generic alerting.
- **FR-014**: The old Codex-facing delivery bridge and `/codex_*` reply UX MUST remain retired.
- **FR-015**: The feature MUST provide machine-readable audit/state records for scheduled and on-demand runs.
- **FR-016**: Deployment and configuration for this feature MUST remain GitOps-managed.

### Key Entities

- **UpstreamReleaseSnapshot**: One normalized view of the official Codex CLI source at a point in time.
- **CodexUpdateSkillState**: Moltis-owned state for latest fingerprint, last delivery outcome, and scheduler checkpoints.
- **CodexUpdateDecision**: The Moltis-native verdict such as `ignore`, `upgrade-later`, `upgrade-now`, or `investigate`.
- **ProjectProfile**: Optional static contract describing how a specific project maps Codex changes to practical recommendations.
- **RecommendationBundle**: Generic or project-specific next steps produced by Moltis.
- **CodexUpdateAuditRecord**: Machine-readable record of one on-demand or scheduled skill run.

### Assumptions & Dependencies

- The new canonical implementation lives in the Moltis project codebase represented by this repository.
- The official Codex changelog remains the primary upstream signal.
- Telegram remains the primary async delivery channel, but the skill must also answer on-demand text requests inside Moltis.
- Project-specific applicability is optional and should be supplied by a stable profile/manifest contract rather than by repo-side runtime ownership.
- Existing repo-side parser/report artifacts may be reused temporarily during migration, but they are not the target architecture.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user can ask Moltis in plain language to check Codex CLI updates and receive a correct Russian answer generated by the Moltis-native skill.
- **SC-002**: Moltis can poll the official source on schedule and emit at most one alert per fresh upstream fingerprint.
- **SC-003**: Telegram delivery no longer depends on the repo-side watcher as the canonical runtime owner.
- **SC-004**: When a valid project profile is present, Moltis includes project-specific recommendations; when absent, Moltis still returns a generic advisory.
- **SC-005**: Operators can inspect one Moltis-owned audit/state record for both scheduled and on-demand runs.
- **SC-006**: The legacy repo-side Codex update bridge remains retired and is not presented as the primary user path.
