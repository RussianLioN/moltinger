# ASC AI Fabrique Demo — План однодневного спринта

**Дата**: 17.03.2026  
**Цель**: Рабочий прототип, демонстрирующий полный пользовательский путь — от описания задачи до генерации one-page summary для кредитного комитета  
**Методология**: Anthropic Playbook (двухфазный спринт за 1 день)  
**Инструмент**: Claude Code (CLI)

---

## Контекст проекта (для понимания)

### Что уже есть
- **UI shell** (index.html + app.css + app.js) — полнофункциональный фронтенд с dark-темой
- **Mock-логика** внутри app.js: 7 discovery-топиков, flow gate→discovery→brief→artifacts
- **API-контракт**: фронтенд шлёт POST `/api/turn`, GET `/api/session`
- **Q&A-спека** (800+ строк) — полная модель фабрики
- **Feature spec** с 4 user stories + BPMN E2E
- **Демо-задача**: генерация one-page summary "ООО Боку до манж" для кредитного комитета

### Что нужно построить за день
Бэкенд-сервер, который:
1. Принимает запросы от существующего UI shell через `/api/turn` и `/api/session`
2. Проводит пользователя через discovery (сбор требований) — используя LLM
3. Формирует brief и показывает его на review
4. После подтверждения brief — генерирует one-page summary по данным клиента
5. Отдаёт результат как артефакт (скачиваемый документ)

## 10 задач для scaffolding (детализация)

### Задача 0: Инициализация проекта
```
Создай Node.js проект:
- package.json с express, anthropic SDK, dotenv, cors
- .env с ANTHROPIC_API_KEY
- Скопируй HTML/CSS/JS в public/
- CLAUDE.md с контекстом проекта
```

### Задача 1: Бэкенд-сервер
```
Express-сервер:
- GET / → отдаёт public/index.html
- POST /api/turn → принимает JSON от фронтенда, пока возвращает mock
- GET /api/session → возвращает состояние сессии
- Формат ответа ТОЧНО совпадает с mockAdapterTurn() из app.js
- In-memory store сессий
```

### Задача 2: LLM-powered discovery
```
Заменить mock-discovery на реальный LLM:
- System prompt = агент-архитектор Moltis
- Контекст: 7 discovery-топиков из MOCK_DISCOVERY_TOPICS
- LLM задаёт следующий вопрос на основе того, что уже покрыто
- Ответ маппится обратно в формат reply_cards
```

### Задача 3: Brief генерация
```
После покрытия всех топиков:
- LLM генерирует структурированный brief
- Brief включает: проблема, пользователи, текущий процесс, входы, выходы, ветвления, метрики
- Brief показывается в side panel как panel-card
- Поддержка действий: confirm_brief, request_brief_correction
```

### Задача 4: One-page summary генератор
```
Четыре LLM-вызова по шаблонам из промпт-документа:
1. ИНФОРМАЦИЯ ПО КЛИЕНТУ (system prompt из секции "Данные по клиенту")
2. ИНФОРМАЦИЯ ПО СДЕЛКЕ (system prompt из секции "Данные по сделке")  
3. ЦЕНООБРАЗОВАНИЕ И ДОХОДНОСТЬ (system prompt из секции "Данные по цене")
4. АНАЛИЗ СОТРУДНИЧЕСТВА И ВКЛАДА (system prompt из секции "Данные по сотрудничеству")
Входные данные: CSV-структура из демо-файла "Боку до манж"
```

### Задача 5: Сквозной pipeline
```
Связать: discovery → brief_confirmed → запуск генерации summary
- При confirm_brief: статус → "в производстве"
- Запускаются 4 параллельных LLM-вызова
- Результат собирается в единый документ
- Артефакт регистрируется в download_artifacts
```

### Задача 6: Демо-данные
```
- Предзагрузить CSV-данные клиента "Боку до манж" как демо-входные данные
- Создать example-prompt для быстрого старта: 
  "Автоматизировать подготовку one-page summary по клиенту для кредитного комитета"
- Прогнать полный happy-path и убедиться, что UI корректно отображает каждый этап
```

### Задача 7: Артефакты и скачивание
```
- Эндпоинт GET /api/artifact/:id → скачивание файла
- Форматирование summary как Markdown-документ
- Side panel показывает артефакты с кнопкой "Скачать"
- Минимальные правки в app.js для реального скачивания (не mock)
```

### Задача 8: Error states
```
- Таймаут LLM-вызова → сообщение "Агент думает дольше обычного..."
- Ошибка API → fallback на mock с пометкой
- Пустой ввод пользователя → мягкое напоминание
- Невалидный access token → корректное сообщение
```

### Задача 9: Финальный прогон
```
- Полный сценарий от gate до скачивания summary
- Проверка всех переходов в UI
- Фиксация демо-ready состояния
```

## Ключевой архитектурный принцип

**Минимум изменений в существующем фронтенде.**  
Фронтенд уже имеет полный UI и контракт. Бэкенд должен возвращать ответы в ТОЧНО ТОМ ЖЕ формате, что и `mockAdapterTurn()` в app.js. Это критически важно — не нужно переписывать фронт, нужно заменить mock бэкендом.

## Демо-сценарий для показа

1. Открыть браузер → gate screen → ввести demo token
2. Нажать пример "one-page summary по клиенту"
3. Агент-архитектор задаёт 4-5 уточняющих вопросов
4. Пользователь отвечает, загружает CSV с данными клиента
5. Discovery завершается → brief на review в side panel
6. Пользователь подтверждает brief
7. Статус → "в производстве" → генерация summary
8. Артефакт готов → скачивание one-page summary
9. Документ содержит 4 секции: клиент, сделка, цена, сотрудничество
