# RCA: Пропуск Speckit-артефактов при выполнении browser-compatibility задачи

**Дата:** 2026-03-06
**Статус:** Resolved
**Влияние:** Временное расхождение между фактическим исполнением задачи и состоянием Speckit-артефактов в ветке; риск потери прозрачности прогресса.
**Контекст:** Ветка `001-browser-compatibility-fix`, issue `moltinger-vt0`

## Ошибка

При выполнении фикса браузерной совместимости изменения в runtime/config были внесены раньше, чем в git была зафиксирована директория `specs/001-browser-compatibility-fix/` с планом и задачами Speckit. Пользователь справедливо указал на несоответствие процесса.

## Анализ 5 Почему

| Уровень | Вопрос | Ответ |
|---------|--------|-------|
| 1 | Почему Speckit-папка не была обработана сразу? | Фокус был на оперативном техфиксe (`docker-compose.prod.yml`) и диагностике прода, а untracked-артефакты Speckit были отложены. |
| 2 | Почему Speckit-артефакты были отложены? | После изменений в параллельной сессии было опасение смешать generated-материалы со срочным фиксом до подтверждения пользователя. |
| 3 | Почему это привело к процессному разрыву? | В рабочем ходе не было обязательного шага reconcile для `specs/*` перед коммитом прикладных изменений. |
| 4 | Почему не было обязательного reconcile шага? | Существующие guard-меры покрывали branch/worktree drift, но не покрывали контроль полноты Speckit-артефактов. |
| 5 | Почему guard-меры неполные? | Definition of Done для Speckit-flow не был формализован как обязательная pre-commit/pre-push проверка наличия/синхронизации spec-пакета. |

## Корневая причина

Неполный процессный контроль: есть защита от drift ветки/ворктри, но нет явной защиты от рассинхрона между исполнением задачи и артефактами Speckit (`spec/plan/tasks/checklists`).

## Принятые меры

1. **Немедленное исправление:** директория `specs/001-browser-compatibility-fix/` добавлена в git и запушена отдельным коммитом.
2. **Предотвращение:** зафиксирован паттерн: при работе по Speckit-задаче сначала проверять и синхронизировать `specs/<feature>/tasks.md`, затем вносить runtime-фиксы.
3. **Документация:** создан текущий RCA отчёт в `docs/rca/`.

## Связанные обновления

- [x] Speckit-пакет `specs/001-browser-compatibility-fix/*` добавлен в ветку
- [x] Прогресс в `tasks.md` обновлён по фактически выполненным задачам
- [ ] Добавить автоматический guard для Speckit completeness (отдельная задача)
- [ ] Обновить общий чеклист сессии pre-flight шагом для `specs/*`

## Уроки

1. **Если задача ведётся по Speckit, `specs/<feature>/` — это не «вторичные файлы»**, а обязательный источник статуса выполнения.
2. **Guard по ветке/ворктри недостаточен без guard по артефактам процесса** — нужен отдельный контроль для `spec/plan/tasks`.
3. **При параллельных сессиях сначала reconcile репозиторный контекст, затем прикладные фиксы**, иначе появляется обоснованная путаница у пользователя.
