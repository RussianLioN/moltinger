# RCA: Повторное подтверждение brief скрывалось из ленты после второй правки

Date: 2026-03-20  
Feature: `024-web-factory-demo-adapter`  
Target: `https://demo.ainetic.tech`

## Ошибка

После второй правки brief агент возвращал тот же текст подтверждения, но UI не показывал его в ленте.  
Для пользователя это выглядело как зависание сценария на шаге review.

## 5 Why

1. Почему пользователь видел зависание?
   - Потому что после отправки второй правки не появлялось новое агентское сообщение в ленте.
2. Почему сообщение не появлялось?
   - Фронтенд отбрасывал сообщение как «дубликат».
3. Почему дедуп классифицировал его как лишний дубликат?
   - Проверка `hasRecentSameAgentBody` удаляла любой одинаковый `body` в последних сообщениях.
4. Почему это было неверно для brief-loop?
   - В review-loop одинаковая формулировка подтверждения может быть корректной и ожидаемой после каждого нового user-turn.
5. Почему ошибка дошла до боя?
   - Не было явного E2E-критерия на повторяемый confirmation-паттерн (две последовательные правки brief с одинаковым ответом агента).

## Root Cause

Лишняя эвристика дедупа во фронтенде:
- сравнение только по тексту агентского `body` в коротком окне сообщений,
- без учёта того, что между двумя одинаковыми ответами был новый пользовательский ход.

## Исправление

Удалён агрессивный блок `hasRecentSameAgentBody` в:
- `web/agent-factory-demo/app.js`
- `asc-demo/public/app.js`

Оставлены безопасные фильтры:
- contiguous duplicate signature,
- `isDuplicateAgentQuestion` (не режет повтор, если после предыдущего такого же текста уже был user-turn).

## Проверка

На боевом `demo.ainetic.tech` повторён сценарий:
1. discovery до `brief review`,
2. правка #1 -> агент подтверждает,
3. правка #2 -> агент снова подтверждает (сообщение теперь отображается),
4. confirm brief -> `Concept pack готов`,
5. проверены preview и скачивание артефактов.

## Профилактика

1. Добавить в регрессионный E2E-кейс обязательную ветку: «две подряд правки brief с одинаковым confirmation body».
2. Для дедуп-логики сообщений использовать правило «допустим повтор, если между одинаковыми агентскими сообщениями был user-turn».
3. В UX-acceptance фиксировать отдельный чек «нет ложного зависания после brief correction».

## Уроки

1. Дедуп в диалоговых системах нельзя строить только на одинаковом тексте; нужен контекст очередности ролей.
2. Повтор агентского текста в review-loop может быть нормальным и не должен автоматически скрываться.
3. Для UX-критичных шагов (brief review/confirm) нужны обязательные боевые E2E-повторы, а не только локальные unit/contract проверки.
