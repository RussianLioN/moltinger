Дата исследования: 2026-03-08  
Локально проверенная версия CLI: `codex-cli 0.111.0`  
Язык документа: русский  
Фокус: официальные возможности Codex CLI, практика сообщества, параллельная работа, оркестрация, фреймворки и подробная инструкция для сложных проектов

## 1. Зачем этот документ

Этот документ нужен не для знакомства с Codex CLI на уровне "как запустить чат в терминале", а для проектной эксплуатации инструмента. Речь идет о сценариях, где у вас есть реальный репозиторий, несколько веток разработки, отдельные рабочие деревья, живая CI/CD-среда, внешние сервисы через MCP, собственные repo-specific инструкции, формализованный code review, задачи, которые занимают не один запрос, а часы или дни, и необходимость держать несколько параллельных направлений работы под контролем.

По состоянию на 8 марта 2026 года Codex CLI уже нельзя рассматривать как просто консольный интерфейс к модели. Это агентная среда разработки с несколькими слоями управления:

1. Интерактивный режим с TUI.
2. Неинтерактивные режимы для запуска задач и code review.
3. Конфигурационный слой через `~/.codex/config.toml`, проектные профили и дополнительные правила.
4. Репозиторный слой инструкций через `AGENTS.md`.
5. Навык-слой через Skills (`SKILL.md`).
6. Интеграционный слой через MCP.
7. Агентный слой через multi-agent и делегирование подзадач.
8. Безопасностный слой через sandbox и approval policies.
9. Автоматизационный слой через JSONL output, output schema, GitHub Actions, app-server и экспериментальные облачные возможности.

Если свести исследование к одной ключевой мысли, то она будет такой: Codex CLI лучше всего работает на сложных проектах не тогда, когда вы пытаетесь "умнее формулировать запросы", а тогда, когда вы строите вокруг него правильную операционную систему работы. Эта система обычно включает:

- структурированные артефакты планирования;
- жесткие правила для агента;
- проектную память в репозитории;
- изоляцию параллельной работы через worktrees;
- предсказуемую модель approval/sandbox;
- стандартизированный запуск повторяемых сценариев;
- разделение ролей между разными сессиями или субагентами;
- обзорный контроль человеком в точках принятия решений.

Именно поэтому ниже я рассматриваю Codex CLI не как "чат", а как инженерный runtime для сложных задач.

## 3. Короткий вывод на одной странице

Если у вас нет времени читать весь документ, вот сжатая версия.

Codex CLI в 2026 году лучше всего использовать для сложных проектов по следующей схеме:

1. Держать общие правила проекта в `AGENTS.md`.
2. Держать повторяемые специализированные workflow в Skills.
3. Использовать `config.toml` и профили для разных режимов работы: исследование, кодинг, ревью, CI.
4. Все внешние интеграции вести через MCP.
5. Большие задачи разбивать на спецификацию, дизайн, задачи и только затем переходить к реализации.
6. Параллельную работу по разным направлениям почти всегда выносить в отдельные git worktrees.
7. Внутрисессионный multi-agent использовать для bounded subtasks: исследование, локальные поиски, независимые куски кода, проверки.
8. Не строить критическую автоматизацию на парсинге TUI; для этого использовать `codex exec`, JSONL, output schema, app-server, GitHub Action и другие headless surface.
9. После каждого обновления CLI перепроверять approval/sandbox behavior и список feature flags, потому что зрелость возможностей различается.
10. Рассматривать Codex как часть инженерного конвейера, а не как "еще одного программиста в терминале".

Главный практический паттерн сообщества выглядит так:

`spec-first -> instructions-first -> worktree-first -> automation-first -> review-gated execution`

Именно он дает на сложных проектах наименьший хаос и наибольшую воспроизводимость.

## 4. Что такое Codex CLI в текущем состоянии

Локально проверенная версия в этой среде: `codex-cli 0.111.0`.

По проверке `codex --help` и help по подкомандам текущая продуктовая поверхность CLI включает:

- основной интерактивный запуск `codex`;
- `exec` для неинтерактивных прогонов;
- `review` для неинтерактивного code review;
- `login` и `logout`;
- `mcp` для управления внешними MCP-серверами;
- `mcp-server` для запуска самого Codex как MCP-сервера;
- `app-server` как экспериментальный серверный surface;
- `app` для запуска Codex Desktop;
- `completion` для shell completion;
- `sandbox` для запуска команд в sandbox runtime;
- `debug`;
- `apply` для применения последнего diff локально;
- `resume` и `fork` для продолжения и ветвления сессий;
- `cloud` как экспериментальный surface для Codex Cloud;
- `features` для feature flags.

Уже один этот список показывает важный сдвиг. CLI перестал быть "одной программой с одним режимом". Он фактически стал мультиинтерфейсной системой, где один и тот же агент может быть использован:

- вручную из терминала;
- из оркестратора;
- из CI;
- через app-server;
- через MCP;
- из desktop-клиента;
- из облачного workflow;
- из инструментов вокруг сессий и history.

Для сложных проектов это значит, что вопрос "как пользоваться Codex CLI" нужно заменять вопросом "какой surface использовать для какого класса задач".

### 4.1. Поверхности использования

Самый полезный способ понимать Codex CLI на практике:

#### Интерактивный TUI

Лучше всего подходит для:

- исследовательской работы;
- анализа кодовой базы;
- поэтапной реализации;
- переговоров с агентом по архитектуре;
- сценариев, где человеку нужно вмешиваться и корректировать курс.

Плюсы:

- естественный диалоговый ритм;
- удобно работать с контекстом;
- легко корректировать агент в процессе;
- хорошо подходит для длинных инженерных сессий.

Минусы:

- не лучший surface для строгой автоматизации;
- сложнее внешняя оркестрация;
- параллельные сценарии быстро становятся шумными без внешней дисциплины.

#### `codex exec`

Лучше всего подходит для:

- одноразовых задач;
- CI/CD;
- автоматизированных локальных прогонов;
- задач, где нужен JSONL output;
- строго формализованного финального вывода;
- серверной оркестрации.

Плюсы:

- headless;
- можно читать stdin;
- можно писать output в файл;
- можно задавать output schema;
- хорошо комбинируется с shell automation.

Минусы:

- слабее интерактивная управляемость;
- для больших неоднозначных задач обычно хуже TUI.

#### `codex review`

Лучше всего подходит для:

- review относительно base branch;
- review uncommitted changes;
- review конкретного commit;
- интеграции в pre-merge checks.

Плюсы:

- отдельный surface под code review;
- хорошо работает как quality gate;
- поддерживает review against base branch.

Минусы:

- это не полноценная замена человеческому ревью;
- сильнее всего полезен как дополнительный reviewer.

#### `mcp-server` и `app-server`

Лучше всего подходят для:

- интеграции Codex в другие инструменты;
- построения собственных control plane;
- headless integrations;
- bridge-сценариев.

Это уже уровень не "пользователь пользуется Codex CLI", а "команда строит систему поверх Codex CLI".

#### `cloud`

Экспериментальный surface. Важно относиться к нему как к evolving capability, а не как к полностью стабильной основе процессов, пока конкретный сценарий не подтвержден в вашей среде.

## 5. Официальные возможности Codex CLI: что реально есть сейчас

В этом разделе я свожу вместе документацию OpenAI и локальную проверку CLI.

### 5.1. Базовый интерактивный режим

Официальный CLI surface предполагает, что если вы просто запускаете `codex`, вы попадаете в интерактивный агентный режим. Через глобальные опции можно сразу задать:

- модель;
- профайл;
- sandbox mode;
- approval policy;
- корневую директорию;
- дополнительные writable directories;
- image input;
- web search;
- feature flags;
- режимы наподобие `--full-auto`.

Это важно потому, что даже "простая" интерактивная сессия на сложном проекте почти никогда не должна стартовать со значений по умолчанию без мысли. В командной эксплуатации безопаснее считать, что каждая длинная сессия Codex должна быть явно запущена в одном из стандартизованных режимов:

- исследование;
- реализация;
- ревью;
- инцидентный hotfix;
- CI/headless;
- многозадачный orchestration profile.

Официально CLI допускает выбор профиля через `-p, --profile`, а также переопределение любых настроек через `-c key=value`. Это делает конфигурацию composable. Для сложных проектов это одна из главных возможностей, потому что вам не нужно завязываться на единственный универсальный конфиг.

### 5.2. Сессии, продолжение работы и форки

Официальные команды `resume` и `fork` оказываются намного важнее, чем кажется новичку.

`resume` нужен, когда вы хотите продолжить уже существующую сессию. Это сохраняет контекст и историю решений. Для длинных задач это часто лучше, чем начинать заново.

`fork` нужен, когда вы хотите ответвить уже существующую сессию в альтернативное направление. Для сложных проектов это сильнейший инструмент, потому что он позволяет:

- отделять рискованные гипотезы от основного плана;
- быстро сравнивать альтернативные архитектуры;
- вести параллельные ветки мысли без уничтожения базового контекста;
- создавать "what-if" ветки для экспериментов.

На практике `fork` в Codex CLI особенно ценен в трех сценариях:

1. Архитектурная неопределенность: вы хотите протестировать два варианта дизайна.
2. Рискованный рефакторинг: вы хотите сначала проверить альтернативный путь, не ломая основную сессию.
3. Командная работа: нужно дать другому инженеру или будущей себе точку входа в уже разобранную проблему.

Сообщество очень явно показывает, что session history, resume и поиск по прошлым сессиям в реальности становятся инфраструктурой. Это подтверждается популярностью инструментов вроде Agent Sessions и Codexia, которые поверх штатной функциональности строят полноценный session browser.

### 5.3. Неинтерактивный режим

Официальная документация по non-interactive mode и локальная проверка `codex exec --help` показывают, что это уже не второстепенная возможность, а ключевой серверный interface.

Практически значимые возможности:

- запуск с prompt аргументом;
- чтение инструкции из stdin;
- `--json` для JSONL event stream;
- `--output-last-message` для записи финального ответа в файл;
- `--output-schema` для структурированного формата ответа;
- `--ephemeral` для запуска без сохранения session files;
- `--skip-git-repo-check` при работе вне git;
- `--add-dir` для дополнительных writable dirs;
- `--full-auto`;
- явный выбор sandbox/approval policy.

На уровне сложных проектов это позволяет строить три класса автоматизации:

#### Автоматизация задач разработки

Примеры:

- сгенерируй release notes по diff;
- сделай review изменений против `main`;
- проверь, какие миграции затронуты;
- подготовь план тестирования;
- собери архитектурное summary по конкретным модулям.

#### Автоматизация CI

Примеры:

- запустить Codex-review на pull request;
- собрать structured output по проблемам в формате JSON;
- использовать `output schema` как контракт между Codex и пайплайном;
- запускать стандартные non-interactive playbooks.

#### Автоматизация оркестрации

Примеры:

- внешний scheduler запускает Codex jobs;
- внутренняя платформа создает task queue поверх `codex exec`;
- несколько Codex worker-процессов обрабатывают разные репозитории.

Здесь возникает главный operational вывод: если вам нужна воспроизводимость, аудит и интеграция, `codex exec` почти всегда предпочтительнее TUI.

### 5.4. Отдельный режим review

`codex review` заслуживает отдельного внимания. Он показывает, что OpenAI не рассматривает review просто как очередной prompt. Review выделен в отдельный mode, что организационно правильно.

Локально подтверждены опции:

- `--uncommitted`;
- `--base <branch>`;
- `--commit <sha>`;
- `--title <title>`.

Для сложного проекта это позволяет стандартизировать несколько полезных сценариев:

1. Локальный pre-commit или pre-push review.
2. Проверка feature branch против `main`.
3. Review конкретного commit.
4. Review серии uncommitted changes перед ручной чисткой diff.

Чего делать не стоит: превращать `codex review` в абсолютный gate, полностью заменяющий человека. Что стоит делать: использовать его как дополнительный reviewer и источник раннего сигнала.

Правильный operational подход обычно такой:

- сначала Codex-review;
- потом человек-review;
- затем автоматические тесты и линтеры;
- затем merge.

### 5.5. AGENTS.md как репозиторный конституционный слой

Официальная документация OpenAI по `AGENTS.md` — одна из самых важных страниц для сложных проектов. Именно она превращает разовую prompt-магиию в устойчивое поведение агента внутри конкретного репозитория.

Почему `AGENTS.md` критичен:

- агент получает проектные правила без необходимости повторять их в каждом чате;
- разные инженеры получают более предсказуемое поведение Codex;
- можно фиксировать workflow, standards, mandatory checks, архитектурные ограничения, правила коммитов, подход к тестированию и т.д.;
- в монорепах можно детализировать правила по поддиректориям.

Официальный смысл `AGENTS.md` не в том, чтобы туда запихнуть всю документацию проекта. Его роль другая: сформулировать стабильные правила принятия решений и поведения агента.

На сложных проектах хороший `AGENTS.md` обычно отвечает на вопросы:

- как искать задачу и как ее claim-ить;
- как писать изменения;
- что нельзя делать без явного разрешения;
- какие команды обязательны перед завершением;
- как обращаться с грязным git workspace;
- как устроены review expectations;
- где искать дополнительную проектную документацию;
- как оформлять handoff;
- какие существуют локальные инварианты.

Изучение официальной страницы и практики сообщества приводит к четкому выводу: для сложного проекта `AGENTS.md` — это не вспомогательная опция, а минимально необходимый слой управления Codex.

### 5.6. Skills как библиотека специализированных режимов работы

Официальная документация по Skills показывает, что Codex CLI уже поддерживает не только статические инструкции репозитория, но и формализованные специализированные навыки, описанные через `SKILL.md`.

На практике Skills нужны там, где один и тот же сложный workflow повторяется снова и снова. Примеры:

- релизный workflow;
- runbook по инциденту;
- создание миграций;
- аудит зависимостей;
- code review по security;
- генерация changelog;
- поддержка конкретного фреймворка;
- инструкция по работе с внутренним MCP-сервисом.

Ключевая разница между `AGENTS.md` и Skills:

- `AGENTS.md` задает общие правила среды.
- Skills задают специализированные воспроизводимые workflow.

Для больших проектов это дает правильную архитектуру проектной памяти:

- стабильное поведение и guardrails в `AGENTS.md`;
- reusable workflows в Skills;
- deeper reference docs рядом со skill при необходимости;
- внешние tools через MCP.

Практический вывод: если ваша команда более двух раз объясняет Codex один и тот же сложный многошаговый сценарий, это почти всегда означает, что нужно создавать Skill, а не писать еще один длинный prompt.

### 5.7. MCP: внешние инструменты и данные

Официальная страница по MCP и локально проверенные `codex mcp --help`, `codex mcp add --help`, `codex mcp get --help`, `codex mcp-server --help` вместе дают очень важную картину.

Codex CLI поддерживает:

- управление внешними MCP servers через `codex mcp`;
- добавление MCP через `--url` или локальную stdio-команду;
- задание env для stdio MCP servers;
- bearer token env var для streamable HTTP MCP servers;
- запуск самого Codex как MCP-server через `codex mcp-server`.

Для сложных проектов значение MCP невозможно переоценить. Через него вы подключаете к агенту:

- документацию;
- internal APIs;
- issue trackers;
- design tools;
- db introspection;
- deployment systems;
- observability;
- proprietary knowledge bases;
- внутренние runbooks.

На практике MCP решает главную проблему длинных инженерных сессий: агент не должен угадывать контекст, если его можно получить как инструментально доступный источник правды.

Хороший зрелый подход для команды выглядит так:

1. Не копировать внутрь prompt большие справочники.
2. Не надеяться на память модели по внутренним системам.
3. Давать Codex доступ к источникам истины через MCP.
4. Ограничивать surface каждого MCP по реальной необходимости.
5. Документировать MCP use cases в Skills и `AGENTS.md`.

### 5.8. Multi-agent

Официальная страница OpenAI по multi-agent и локально наблюдаемый feature flag `multi_agent` в статусе `experimental` показывают, что Codex CLI уже умеет внутреннюю делегацию подзадач. Это критически важный слой для сложных проектов, но использовать его нужно трезво.

Multi-agent — не магическая кнопка "ускорить все". Это механизм распараллеливания bounded tasks.

Типовые роли, которые хорошо подходят под делегирование:

- исследователь/эксплорер кода;
- ревьюер;
- узкоспециализированный worker;
- помощник по тестам;
- классификатор входящих данных;
- подготовщик structured output;
- sidecar-аудитор.

Типовые задачи, которые плохо подходят под делегирование:

- срочная блокирующая задача, от результата которой зависит следующий локальный шаг;
- расплывчатое архитектурное решение без четких границ;
- сложный рефакторинг с высокой связностью между файлами;
- задачи, где сложно развести ownership.

Официальные паттерны multi-agent особенно полезны в таких сценариях:

1. Параллельный кодовый поиск по разным подсистемам.
2. Параллельное чтение документации и локального кода.
3. Независимые дискретные правки в разных модулях.
4. Отдельный review после основной реализации.
5. Fan-out по CSV или по набору независимых входов.

Но есть принцип, который нельзя нарушать: multi-agent внутри одной сессии не заменяет git isolation. Если два независимых направления могут конфликтовать по файлам, их надежнее разносить по worktrees, а не только по субагентам.

### 5.9. Config, profiles и project-level настройка

Официальные страницы `Config basics`, `Advanced config`, `Config reference` дают тот слой, без которого Codex CLI в большом проекте быстро превращается в хаос.

Ключевая мысль: сложный проект не должен жить на одном глобальном config без профилей и без project-specific правил.

Что нужно понимать:

- глобальный конфиг в `~/.codex/config.toml` нужен для пользовательских defaults;
- project-level конфиг нужен для repo-specific behavior;
- profiles нужны для режимов работы;
- часть advanced behavior настраивается не промптами, а конфигом;
- feature flags должны восприниматься как управляемый runtime surface.

Практически полезные идеи:

#### Профили

Создайте разные профили под разные задачи. Например:

- `research`;
- `implementation`;
- `review`;
- `ci`;
- `hotfix`;
- `unsafe-local-lab` для локальных изолированных экспериментов.

Плюс профилей в том, что вы стандартизируете запуск. Если в команде все говорят "запусти Codex в профиле review", это уже операционная дисциплина.

#### Project-specific настройки

Отдельные репозитории могут требовать:

- другие правила approval;
- другие writable dirs;
- отдельные MCP;
- другие ограничения по проектным документам;
- включенные или выключенные фичи;
- разные default models.

#### Advanced config

По официальным страницам важны темы:

- notifications;
- history/session behavior;
- projects;
- metrics/analytics;
- rules/requirements;
- managed configuration;
- MCP configuration;
- модельные настройки и speed trade-offs.

Именно advanced config чаще всего недооценивают. На больших проектах он становится operational glue, который убирает лишние ручные действия.

### 5.10. Rules и requirements

Официальные страницы `Rules` и связанные материалы показывают еще один уровень governance. По сути это means of expressing operational guardrails вне простого prompt текста.

На практике правила нужны там, где вы хотите:

- запретить определенные команды;
- заранее разрешить безопасные шаблоны;
- фиксировать policy по approvals;
- задавать системные ограничения;
- снизить случайность поведения.

Если `AGENTS.md` отвечает на вопрос "как в целом должен вести себя агент в этом репозитории", то rules/requirements отвечают на вопрос "какие ограничения и разрешения действуют на runtime-уровне".

Для сложных проектов это особенно полезно при:

- работе в production-like окружениях;
- orchestration большого числа сессий;
- делегировании работы менее опытным инженерам;
- интеграции Codex в CI;
- работе с критичными командами и секретами.

### 5.11. Approval policy и sandbox

Официальная security-документация и локально подтвержденные CLI опции дают важную матрицу поведения:

Sandbox modes:

- `read-only`;
- `workspace-write`;
- `danger-full-access`.

Approval policies:

- `untrusted`;
- `on-request`;
- `never`;
- `on-failure` как deprecated.

Плюс в локальной CLI отдельно присутствует `codex sandbox` с OS-specific runtime:

- `macos`/Seatbelt;
- `linux`/Landlock+seccomp;
- `windows` restricted token.

Для сложных проектов именно этот слой определяет, насколько безопасно и предсказуемо поведет себя агент при доступе к shell. Практически это один из самых критичных operational decisions.

Я рекомендую мыслить не "какой режим удобнее", а "какой риск-профиль у сценария".

Пример:

- анализ кода без записи: `read-only` + `untrusted`;
- типовая реализация в локальном репо: `workspace-write` + `on-request`;
- headless CI в изолированном runner: часто `workspace-write` + `never`;
- полностью внешне sandboxed ephemeral container: иногда допускается более агрессивный режим, но только если вы уверены в инфраструктурной изоляции.

Одна из самых полезных community-мыслей здесь: approval/sandbox contract нужно тестировать после апдейтов CLI. Нельзя предполагать, что semantics не меняются.

### 5.12. Feature flags и зрелость возможностей

Локально проверенный `codex features list` показывает, что в текущем runtime у фич есть разные стадии:

- `stable`;
- `experimental`;
- `under development`;
- `deprecated`;
- `removed`.

На 0.111.0 среди заметных состояний:

- `multi_agent` — experimental;
- `js_repl` — experimental;
- `fast_mode` — stable;
- `shell_tool` — stable;
- `shell_snapshot` — stable;
- `unified_exec` — stable;
- `sqlite` — stable;
- `prevent_idle_sleep` — experimental;
- ряд фич находится в `under development` или уже `removed`.

Для сложных проектов это значит:

1. Не все, что "есть", одинаково пригодно как фундамент процесса.
2. Стабильные фичи можно закладывать в командный workflow гораздо увереннее.
3. Экспериментальные фичи нужно инкапсулировать, документировать и проверять на вашем стекe.
4. Removed/deprecated surface нельзя использовать в долгоживущих playbooks.

Хороший operational стиль: завести внутри команды свой compatibility table по используемым фичам Codex CLI и обновлять его после upgrade.

### 5.13. App server, desktop app и cloud

Локально подтвержденные surfaces `app`, `app-server`, `cloud` показывают, что OpenAI двигает Codex не только как терминальный tool, но как платформу с несколькими потребителями интерфейса.

Что это значит для сложных проектов:

- можно строить собственные control plane поверх Codex;
- уместно проектировать внутренние dashboard и orchestration UI;
- реальный рабочий сценарий может быть гибридным: часть работы в TUI, часть в app, часть через server integration;
- CLI history и app-level UX со временем будут сильнее сближаться.

Но operational вывод должен быть прагматичным: пока какой-то surface отмечен как experimental или мало проверен у вас в команде, не надо делать его единственным критичным каналом.

- [Read-only approval regression issue](https://github.com/openai/codex/issues/3684).

Из них вырисовывается несколько удивительно согласованных паттернов.

### 6.1. Почти все зрелые практики сходятся к spec-first

Самый устойчивый community-сигнал не в том, что людям нужен "более умный агент". Людям нужен агент, который сначала помогает формализовать требования, дизайн и задачи, и лишь потом начинает писать код.

Причина простая. На больших проектах главная проблема редко находится на уровне синтаксиса. Основная проблема — это:

- неясная граница задачи;
- скрытые зависимости;
- конфликтующие предположения;
- размытые acceptance criteria;
- невыявленные риски изменения поведения.

Поэтому community frameworks вроде `cc-sdd` получают высокий спрос: они не "усиливают модель", а дисциплинируют процесс.

Практический вывод:

Если ваша команда уже чувствует, что Codex иногда "слишком быстро прыгает к реализации", проблема, скорее всего, не в том, что агент плохой. Проблема в том, что у вас отсутствует явный spec-first workflow.

### 6.2. Почти все зрелые практики сходятся к worktree-first

Вторая по силе мысль сообщества: параллельную работу лучше всего изолировать через git worktrees.

Почему не просто несколько сессий в одном рабочем дереве:

- агенты начинают конфликтовать по файлам;
- трудно понять, кто что поменял;
- сложно откатывать отдельные направления;
- ревью смешивается;
- контекст переплетается;
- повышается риск затереть пользовательские изменения.

Почему worktrees работают:

- каждый агент получает отдельную ветку;
- файловая изоляция физическая, а не только логическая;
- merge/rebase становятся управляемыми;
- можно отдельно тестировать каждое направление;
- удобно держать разные IDE, tmux panes или desktop windows.

По сути worktree — это главный примитив параллельной работы поверх Codex CLI. Не multi-agent, не GUI, не scheduler, а именно worktree.

### 6.3. История сессий — это уже инфраструктура, а не бонус

Популярность Agent Sessions, Codexia и похожих инструментов показывает, что людям критично:

- искать по старым сессиям;
- быстро возобновлять контекст;
- повторно использовать удачные prompt patterns;
- видеть историю решений и трассу расследования;
- не терять многочасовые исследовательские прогонки.

Это очень важный operational вывод. Если вы ведете серьезную работу с Codex CLI, то `~/.codex/sessions` и смежный session layer нужно воспринимать как полезный артефакт системы, а не как временный мусор.

Практически это означает:

- не удалять историю без причины;
- иметь tooling для поиска по прошлым сессиям;
- использовать `resume` и `fork` осознанно;
- для команды хранить handoff context в репозитории, а не только в session logs.

### 6.4. Для серьезной автоматизации нужен headless surface

Сообщество явно показывает неудовлетворенность сценариями, где приходится оркестрировать интерактивный TTY как будто это API. Отсюда интерес к:

- `codex exec`;
- JSONL event streams;
- output schema;
- app-server;
- SDK-оберткам;
- headless adapters;
- desktop control planes, работающим поверх формальных интерфейсов.

Operational lesson:

Если ваша команда собирается строить scheduler, queue, supervisor, dashboard, CI integration или agent farm, не стоит закладывать архитектуру на "мы просто будем скриптом парсить то, что рисует TUI". Это хрупкий путь.

### 6.5. Нужны lifecycle hooks и уведомления

Популярность discussion о hooks/notifications говорит о банальной, но важной вещи: длинная агентная работа без уведомлений неудобна.

В реальной эксплуатации вы не хотите каждые 30 секунд смотреть, не закончилась ли сессия. Поэтому notifications, hook-like behavior, интеграция с OS signals или desktop surfaces становятся не косметикой, а productivity multiplier.

### 6.6. Люди хотят specialization, а не одного "мега-агента"

Обсуждения subagents и специализации показывают зрелое понимание процесса. Сообщество хочет:

- отдельного reviewer;
- отдельного architect;
- отдельного debugger;
- отдельного docs-writer;
- отдельного test-writer;
- отдельного security-minded runner.

Это полностью совпадает с тем, что эффективно на сложных проектах. Один универсальный агент удобен только на маленьких задачах. На больших задачах specialization почти всегда полезнее.

## 7. Самые популярные фреймворки и обвязки вокруг Codex CLI

Здесь важно договориться о терминах. У Codex CLI нет единственного "официального фреймворка для сложных проектов" в том смысле, как у веб-разработки есть Next.js или Django. Вместо этого вокруг Codex CLI формируется экосистема паттернов и инструментов. Некоторые из них ближе к framework, некоторые ближе к control plane, некоторые ближе к workflow kit.

Ниже я перечисляю наиболее заметные категории.

### 7.1. Spec-driven workflow frameworks

#### `cc-sdd`

Источник: [gotalab/cc-sdd](https://github.com/gotalab/cc-sdd)

Это самый сильный по traction community-framework в этой выборке. Его главная идея — дисциплинировать AI-assisted development через pipeline:

- requirements;
- design;
- tasks;
- implementation.

Почему он важен именно для Codex CLI:

- снимает проблему преждевременного кодинга;
- помогает дробить задачи;
- хорошо сочетается с несколькими агентами и worktrees;
- создает общие артефакты, понятные человеку и агенту;
- делает командную работу устойчивее.

Если у вас сложный проект, из всех community решений именно этот класс стоит изучать первым.

### 7.2. Desktop orchestration / command center

#### `Codexia`

Источник: [milisp/codexia](https://github.com/milisp/codexia)

Codexia — это пример того, что в community быстро возникает потребность в "рабочем столе" для Codex CLI. Не просто терминал, а command center с:

- file tree;
- prompt notepad;
- session history;
- git worktree support;
- diff viewer;
- preview вспомогательных файлов;
- management surface для сессий.

Это особенно полезно, если в команде часть людей продуктивнее работает не из чистого TUI, а из визуальной панели.

#### `Commander`

Источник: [autohandai/commander](https://github.com/autohandai/commander)

Commander идет еще дальше и показывает паттерн "оркестратор нескольких coding CLI agents" с worktree-based isolation. Это хороший образец того, как можно организовать несколько параллельных агентных потоков в едином UI.

#### `CodexFlow`

Источник: [lulu-sk/CodexFlow](https://github.com/lulu-sk/CodexFlow)

CodexFlow фокусируется на conversation management, history aggregation и multi-session работе, в том числе в WSL/PowerShell context. Это особенно интересно для mixed-platform команд.

### 7.3. Session observability and search

#### `Agent Sessions`

Источник: [jazzyalex/agent-sessions](https://github.com/jazzyalex/agent-sessions)

Если у вас активная работа с несколькими агентами и длинными сессиями, то session browser становится почти так же важен, как IDE. Agent Sessions показывает этот слой очень наглядно:

- поиск по старым сессиям;
- resume;
- analytics;
- read-only indexing;
- history-based reuse.

Такие инструменты полезны не потому, что Codex чего-то "не умеет", а потому, что на масштабе сессий становится много и ими нужно управлять.

### 7.4. Headless orchestration and SDK layers

#### `headless-coder-sdk`

Источник: [OhadAssulin/headless-coder-sdk](https://github.com/OhadAssulin/headless-coder-sdk)

Это уже шаг в сторону platform engineering. Главная идея — стандартизировать threads, streaming, structured outputs и sandboxed interaction для нескольких AI coders, включая Codex.

Для сложных проектов особенно полезны такие идеи:

- server-side adapters;
- unified thread abstraction;
- cooperative cancellation;
- structured outputs;
- параллельные reviewer agents;
- централизованный orchestration layer.

Если вы строите внутреннюю платформу, а не просто пользуетесь CLI руками, именно этот класс решений становится особенно ценным.

### 7.5. Session farms и terminal-first orchestration

#### `codex-cli-farm`

Источник: [waskosky/codex-cli-farm](https://github.com/waskosky/codex-cli-farm)

Это очень показательный пример terminal-native orchestration через `tmux`. Он иллюстрирует важный принцип: если у вас нет желания или возможности строить GUI, можно сделать manager layer поверх tmux sessions, логов и restore/snapshot behavior.

Такой подход особенно хорош для:

- SSH-first команд;
- long-running remote workers;
- homelab/server setups;
- агентных ферм;
- инфраструктурного engineering mindset.

## 8. Как правильно организовывать сложный проект в Codex CLI

Теперь переходим к самому практическому разделу: как именно строить рабочую систему.

Я дам сначала общие принципы, затем конкретный operating model.

### 8.1. Принцип 1: Codex должен входить в проект как инженерная роль, а не как универсальный чат

Самая частая ошибка на больших проектах — использовать Codex так, как будто это бесконечно терпеливый чатбот, которому можно каждый раз заново пересказывать репозиторий.

Это плохо по нескольким причинам:

- теряется воспроизводимость;
- каждый инженер пишет свои версии правил;
- решения плавают между сессиями;
- высок риск забыть важные ограничения;
- сложно масштабировать практику на команду.

Правильный подход:

- проект определяет общий слой инструкций;
- повторяемые workflow вынесены в reusable формы;
- разные режимы работы стандартизированы профилями;
- агенту редко приходится "угадывать", что здесь принято.

### 8.2. Принцип 2: сначала договоритесь о слоях управления

Хорошая production-like структура обычно выглядит так:

#### Слой 1. Глобальный пользовательский config

Что здесь хранить:

- личные defaults;
- удобные профили;
- notifications;
- базовые MCP;
- пользовательские preferences;
- settings, не зависящие от конкретного репозитория.

#### Слой 2. Project-level config

Что здесь хранить:

- project-specific profiles;
- ограничение writable dirs;
- project MCP;
- особые feature toggles;
- локальные overrides.

#### Слой 3. `AGENTS.md`

Что здесь хранить:

- правила поведения агента в репозитории;
- обязательные workflow;
- запреты и ограничения;
- требования к тестам, review и handoff;
- указатели на внутренние документы.

#### Слой 4. Skills

Что здесь хранить:

- сложные повторяемые workflow;
- domain-specific playbooks;
- инструкции по конкретным интеграциям;
- reusable operational knowledge.

#### Слой 5. MCP

Что здесь хранить:

- доступ к внешним системам;
- документации;
- данным;
- внутренним API и справочникам.

Если вы не разложили процесс по этим слоям, сложный проект почти наверняка будет страдать от неустойчивого поведения Codex.

### 8.3. Принцип 3: различайте уровни параллельности

Это ключевой раздел, потому что многие смешивают разные виды параллелизма.

#### Уровень A. Параллельное мышление внутри одной сессии

Это multi-agent, subagents, delegated explorer/worker tasks. Хорошо подходит для bounded subtasks.

#### Уровень B. Параллельное исполнение в разных worktrees

Это основной механизм изоляции независимых направлений работы.

#### Уровень C. Параллельная автоматизация через headless Codex

Это `codex exec`, scheduler, CI, app-server, cloud/task runners.

Проблемы начинаются тогда, когда командa пытается один уровень подменить другим. Например:

- вместо worktrees пытается держать пять конкурирующих задач в одном дереве;
- вместо headless API оркестрирует TUI;
- вместо локальной делегации подзадач поднимает лишние ветки и процессы;
- вместо архитектурной спецификации пытается "разрулить по ходу".

## 9. Практика параллельной работы: правильная архитектура

Ниже — рекомендуемая схема для команды, ведущей несколько направлений параллельно.

### 9.1. Базовая схема

Пусть у вас есть один основной репозиторий и четыре направления:

- крупный feature;
- bugfix;
- infra cleanup;
- документация.

Я рекомендую такую организацию:

1. Основной репозиторий в одной директории.
2. Под каждое независимое направление отдельный git worktree.
3. Под каждый worktree своя ветка.
4. Внутри worktree либо один интерактивный Codex, либо один orchestration supervisor и несколько bounded subagents.
5. Для cross-cutting analysis использовать отдельные read-only исследовательские сессии.
6. Для review использовать либо отдельный `codex review`, либо отдельный review-worktree, либо отдельного agent-reviewer.

Почему это хорошо:

- конфликтующие правки не смешиваются;
- проще merge;
- проще rollback;
- проще handoff;
- проще тестировать;
- проще видеть ownership.

### 9.2. Как делить задачи между worktrees и subagents

Используйте такое правило.

Создавайте отдельный worktree, если:

- подзадача имеет отдельный lifecycle;
- ожидается отдельный commit stream;
- высок риск конфликтов по файлам;
- нужен самостоятельный review;
- работа займет больше короткой bounded сессии;
- нужен отдельный инженерный контекст.

Используйте subagent внутри текущей сессии, если:

- задача хорошо ограничена;
- write set локален и понятен;
- вы можете четко задать ownership;
- результат нужен как sidecar, а не как отдельная продуктовая ветка;
- это исследование, аудит или небольшой дискретный change.

### 9.3. Идеальная матрица ролей

Для сложного проекта удобно мыслить ролями:

- `Lead session`: держит главный план, принимает решения, интегрирует результат.
- `Explorer`: ищет по коду, документации, логам, схемам.
- `Worker`: делает bounded implementation в локальном scope.
- `Reviewer`: делает post-implementation проверку.
- `Release/CI session`: запускает headless quality gates и проверяет финальное состояние.

Если работа действительно крупная, эти роли можно распределить по разным worktrees и/или orchestration surfaces.

### 9.4. Правило ownership

Самое важное требование к параллельной работе: у каждой параллельной единицы должен быть свой ownership.

Ownership может быть:

- по директории;
- по модулю;
- по типу задачи;
- по этапу lifecycle;
- по issue.

Плохой пример:

"Ты, Codex, займись фронтендом, бэкендом и миграциями параллельно".

Хороший пример:

- worktree A: UI-only изменения в `apps/web/src/features/billing`.
- worktree B: API-only изменения в `services/api/billing`.
- worktree C: docs-only изменения в `docs/billing`.
- subagent D: исследование влияния на тесты.

## 10. Подробная инструкция: как запускать Codex CLI на сложном проекте

Далее — пошаговая operational инструкция.

### Шаг 1. Разделите конфигурацию на режимы

Сначала определите минимальный набор режимов работы. Я рекомендую как минимум четыре:

- `research`;
- `implementation`;
- `review`;
- `ci`.

Логика такая:

- `research` максимально безопасен и ориентирован на чтение;
- `implementation` допускает запись в workspace;
- `review` заточен под анализ diff;
- `ci` ориентирован на headless reproducibility.

Даже если вы не используете все профили с первого дня, сама идея профилей резко повышает дисциплину.

### Шаг 2. Напишите сильный `AGENTS.md`

Хороший `AGENTS.md` для сложного проекта обычно включает:

#### Что это за проект

Коротко:

- стек;
- архитектурный стиль;
- ключевые подсистемы;
- где лежит основная документация.

#### Как агент должен работать

Например:

- сначала читать код и искать контекст;
- не делать предположений о структуре проекта;
- перед изменением файлов объяснять план;
- использовать быстрые инструменты поиска;
- не трогать чужие изменения;
- не использовать destructive git commands без явного разрешения.

#### Что обязательно перед завершением задачи

Например:

- прогнать нужные quality gates;
- обновить issue status;
- синхронизироваться с трекером;
- закоммитить;
- запушить;
- проверить `git status`;
- сделать handoff summary.

#### Как относиться к грязному workspace

Это особенно важно, потому что Codex работает в реальной файловой системе и должен иметь четкую политику:

- не откатывать чужие изменения;
- при конфликте останавливаться и спрашивать;
- читать файл внимательнее, если там уже есть изменения;
- не делать destructive cleanup.

### Шаг 3. Вынесите повторяемые сценарии в Skills

Если задача повторяется, не храните ее как "любимую формулировку запроса". Сделайте Skill.

Лучшие кандидаты на Skill:

- release workflow;
- dependency upgrade playbook;
- security audit;
- incident response;
- data migration checklist;
- postmortem template;
- working with a specific internal service;
- generating structured reports.

Навык дает две вещи:

1. Сокращение числа токенов на входе.
2. Повышение воспроизводимости между людьми и сессиями.

### Шаг 4. Подключите MCP только по делу

Ошибка — пытаться подключить "все подряд". Это увеличивает complexity surface.

Подключайте MCP по такому принципу:

- есть источник истины, к которому Codex реально должен ходить;
- этот источник часто нужен;
- его лучше читать инструментально, чем копировать текстом;
- доступ можно ограничить разумно;
- у команды есть понятный use case.

Типовой минимальный набор для сложного проекта:

- docs/reference MCP;
- issue tracker MCP;
- deployment/infra MCP;
- observability/logs MCP;
- possibly design/doc database MCP.

### Шаг 5. Отделите planning от coding

Это один из самых важных практических шагов.

Перед тем как дать Codex право писать код на крупной задаче, заставьте процесс пройти хотя бы упрощенные стадии:

1. Сбор контекста.
2. Формулировка задачи и границ.
3. Предлагаемый дизайн.
4. Декомпозиция на steps или issues.
5. Только потом реализация.

Это можно реализовать:

- через отдельный plan mode workflow;
- через `AGENTS.md`;
- через spec-driven framework вроде `cc-sdd`;
- через team discipline.

### Шаг 6. На большие задачи создавайте отдельные артефакты

Если feature крупный, создавайте в репозитории артефакты:

- `spec.md`;
- `plan.md`;
- `tasks.md`;
- `risk-log.md` при необходимости;
- `rollback.md` для рискованных изменений.

Плюсы:

- Codex опирается на явные документы;
- людям проще review-ить замысел;
- легче разбивать задачу на sub-tasks;
- можно делать handoff без переписывания контекста;
- легче параллелить работу по ownership.

### Шаг 7. Используйте worktrees как базовый механизм параллельной изоляции

Практически рекомендуется:

1. Основная ветка остается чистой.
2. Для каждого направления работы создается отдельный worktree.
3. Naming должен быть предсказуемым.
4. В каждой директории — отдельная сессия Codex.

Хороший naming pattern:

```bash
git worktree add ../repo-feature-auth feat/auth
git worktree add ../repo-bugfix-cache fix/cache-invalidation
git worktree add ../repo-docs-api docs/api-overhaul
```

Альтернативно можно держать все worktrees в одном каталоге:

```bash
git worktree add ~/.codex/worktrees/repo-auth feat/auth
git worktree add ~/.codex/worktrees/repo-cache fix/cache-invalidation
```

Главное требование — читаемость.

### Шаг 8. Разделите запуск по surface

Для каждой категории работ используйте подходящий interface:

#### Исследование

Интерактивный `codex` в `read-only` или очень аккуратном режиме.

#### Реализация

Интерактивный `codex` в `workspace-write` и `on-request`.

#### Автоматический review

`codex review --base main` или `codex review --uncommitted`.

#### CI/automation

`codex exec --json` с `output schema`.

#### Внешняя интеграция

`app-server`, `mcp-server`, GitHub Action или обертка/SDK.

### Шаг 9. Используйте multi-agent только там, где это оправдано

Хорошие кейсы:

- параллельно прочитать три разные подсистемы;
- дать одному агенту исследовать документацию, а второму — код;
- дать двум reviewer-агентам проверить разные risk areas;
- fan-out по независимым input items;
- поручить subagent локальную правку в четком ownership scope.

Плохие кейсы:

- дать нескольким агентам одновременно редактировать один и тот же модуль без разграничения;
- надеяться, что они "сами договорятся";
- распараллеливать то, что блокирует следующий локальный шаг;
- использовать subagents вместо нормальной декомпозиции.

### Шаг 10. Стандартизируйте handoff

Любая длительная Codex-задача должна завершаться не только кодом, но и handoff artifact.

Минимальный handoff должен отвечать:

- что сделано;
- что не сделано;
- какие риски остались;
- какие quality gates прогнаны;
- какие команды запускались;
- где смотреть diff;
- что следующему человеку делать первым.

Это особенно важно при переключении между сессиями, worktrees и людьми.

## 11. Как строить workflow для разных типов сложных задач

Ниже — конкретные operating models по типам работ.

### 11.1. Крупный feature

Рекомендованный pipeline:

1. Исследовательская read-only сессия.
2. Формализация `spec.md`.
3. Дизайн и `plan.md`.
4. Декомпозиция в `tasks.md`/issue tracker.
5. Выделение worktrees по substreams.
6. Реализация по worktrees.
7. Локальный Codex-review.
8. Человеческий review.
9. Merge по частям.

Где помогает multi-agent:

- подготовка вариантов дизайна;
- поиск зависимостей;
- сбор impact analysis;
- post-implementation review.

Где нужен worktree:

- любая существенная ветка реализации.

### 11.2. Большой рефакторинг

Это самый опасный класс задач для агентной работы, потому что:

- высокая связность;
- трудно оценить blast radius;
- легко испортить поведение без изменения тестов;
- diff быстро разрастается.

Рекомендуемый подход:

1. Отдельная plan-only фаза.
2. Явный список invariants.
3. Фиксация rollback strategy.
4. Разбиение на smallest safe slices.
5. Один worktree на slice или серию близких slices.
6. После каждого slice — review и тесты.

Главное правило: не давать агенту "сделай весь рефакторинг" в одном заходе.

### 11.3. Инцидент и hotfix

Здесь важны скорость и безопасность.

Рекомендованный flow:

1. Отдельный hotfix profile.
2. Read-only triage сессия.
3. Четко зафиксированная гипотеза причины.
4. Минимальный diff.
5. Отдельный worktree/branch под hotfix.
6. Быстрый review.
7. Деплой.
8. Postmortem и нормализация знаний в repo.

Чего не делать:

- совмещать hotfix с cleanup/refactor;
- позволять агенту "попутно улучшить все рядом";
- работать в сильно permissive режиме без нужды.

### 11.4. Исследование legacy codebase

Здесь Codex CLI особенно силен, если вы правильно ограничиваете запись.

Рекомендованный flow:

1. Запуск в read-only.
2. Подключение docs/issues/logs через MCP.
3. Параллельные explorer subagents по подсистемам.
4. Сбор architectural map.
5. Отдельный исследовательский отчет в репозитории.

Главная цель — не код, а уменьшение неизвестности.

### 11.5. Documentation and migration work

Такие задачи хорошо подходят как для интерактивного, так и для headless режима.

Примеры:

- миграция командного workflow;
- обновление инструкций;
- построение onboarding docs;
- changelog generation;
- policy updates.

Здесь полезно:

- использовать output schema;
- хранить structured artifacts;
- делать отдельный docs-review.

## 12. Что именно писать в `AGENTS.md` для сложного проекта

Ниже — practical template на уровне структуры, а не обязательного текста.

### 12.1. Верхний блок

Нужно указать:

- что за проект;
- какой трекер задач используется;
- где смотреть архитектурные документы;
- какие обязательные workflows действуют.

### 12.2. Общие инженерные правила

Примеры:

- сначала изучай код, потом меняй;
- не делай предположений без проверки;
- не трогай чужие незавершенные изменения;
- не используй destructive команды без явного разрешения;
- перед редактированием файлов дай короткое объяснение;
- после существенной реализации прогоняй quality gates.

### 12.3. Правила по git

Примеры:

- не делать `git reset --hard`;
- не делать `git checkout --` без запроса;
- не amend-ить commit без просьбы;
- не оставлять незапушенную локальную работу как "готово";
- явно проверять `git status`.

### 12.4. Правила по branch/worktree workflow

Примеры:

- каждый крупный stream в отдельном worktree;
- naming pattern branches;
- обязательный rebase/pull policy;
- требования к merge preparation.

### 12.5. Правила по ответам и handoff

Примеры:

- краткий summary сделанного;
- что проверено;
- что не удалось проверить;
- какие риски остались;
- где лежат измененные артефакты.

### 12.6. Правила по безопасности

Примеры:

- работа с секретами;
- какие команды требуют approval;
- что делать при sandbox failures;
- как запрашивать escalation.

### 12.7. Привязка к навыкам и специальным runbooks

Очень полезно прямо ссылаться на Skills и runbooks:

- когда использовать skill X;
- где лежит skill Y;
- как работает release process;
- где искать расследования и RCA.

## 13. Как проектировать Skills для сложного проекта

Хороший Skill — это не "очень длинная инструкция". Хороший Skill — это компактный, запускаемый operational contract.

### 13.1. Когда точно нужен Skill

Создавайте Skill, если:

- workflow повторяется;
- у него есть четкая цель;
- есть стабильная последовательность шагов;
- есть узкоспециализированный домен;
- хочется уменьшить variability;
- есть набор reference files или scripts.

### 13.2. Из чего должен состоять хороший Skill

Минимально:

- короткое описание;
- когда использовать;
- где искать reference files;
- какие scripts запускать;
- какой expected output;
- какие guardrails соблюдать.

### 13.3. Примеры навыков, которые реально окупаются

- release-manager;
- database-migration-auditor;
- security-reviewer;
- changelog-generator;
- postmortem-summarizer;
- dependency-upgrader;
- UI-regression-tester;
- deployment-runbook.

### 13.4. Главная ошибка со Skills

Делать Skills слишком общими. Skill должен быть bounded. Если навык можно применить к чему угодно, он перестает помогать.

## 14. Как использовать MCP в сложном проекте без перегрузки

MCP может как резко усилить Codex, так и утопить систему в complexity.

### 14.1. Какие MCP особенно полезны

#### Documentation MCP

Лучше всего для:

- фреймворков;
- vendor docs;
- internal platform docs;
- model docs;
- SDK docs.

#### Tracker MCP

Лучше всего для:

- Linear/Jira;
- issue ownership;
- status updates;
- linking task to code changes.

#### Ops/Infra MCP

Лучше всего для:

- observability;
- deployment state;
- logs;
- incidents;
- configuration state.

#### Data access MCP

Лучше всего для:

- схемы БД;
- справочники;
- read-only inspection.

### 14.2. Какие MCP опасны без дисциплины

- write-capable ops MCP без сильных правил;
- широкие internal tools без scope restriction;
- интеграции, где агент может совершать дорогостоящие или разрушительные действия без явных guardrails.

### 14.3. Лучший operational подход

1. Начинать с read-only MCP.
2. Сужать surface каждого MCP.
3. Привязывать MCP usage к Skills.
4. Документировать MCP dependencies.
5. Делать review права доступа.

## 15. Как строить надежный headless pipeline с Codex CLI

Если вы хотите реальную автоматизацию, используйте headless-first design.

### 15.1. Базовый шаблон

Для repeatable task:

1. Подготовьте prompt template или Skill.
2. Запускайте `codex exec`.
3. Используйте `--json`, если нужен event stream.
4. Используйте `--output-schema`, если ответ должен быть machine-readable.
5. Пишите final output в файл через `--output-last-message`.
6. Валидируйте downstream consumer.

### 15.2. Где это особенно полезно

- PR review bot;
- release note generator;
- issue triage;
- architecture summarization;
- backlog grooming;
- documentation generation;
- security check explanation;
- flaky test diagnosis summary.

### 15.3. Почему output schema важнее, чем кажется

Без output schema downstream automation быстро начинает зависеть от стилистики ответа модели. Это плохо.

Со schema вы получаете:

- предсказуемость;
- валидацию;
- более безопасную интеграцию;
- меньше brittle parsing.

### 15.4. Почему `--ephemeral` полезен

Для некоторых CI-задач и batch jobs нет смысла копить session artifacts. `--ephemeral` полезен там, где важнее чистота и воспроизводимость, чем история.

## 16. Как комбинировать Codex CLI с GitHub Actions

Официальная документация OpenAI отдельно покрывает GitHub Action. Практический смысл прост: Codex может участвовать в CI как formalized reviewer/assistant.

Я рекомендую следующие safe use cases:

- review pull request diff;
- summarize risky changes;
- generate testing checklist;
- triage failing workflow logs;
- produce structured report to attach to PR.

Менее безопасные use cases:

- auto-merge based only on Codex judgement;
- прямое автоматическое применение правок в основной ветке без review;
- широкие write permissions в production-sensitive workflows.

Если включаете Codex в GitHub Actions, обязательно задайте:

- минимально необходимые permissions;
- четкую approval/policy модель;
- структурированный output;
- human review checkpoint.

## 17. Безопасность: практическая модель для команд

На сложных проектах безопасность — не только про секреты. Это про control of agency.

### 17.1. Разделяйте доверие по средам

Для local dev:

- чаще всего `workspace-write` + `on-request`.

Для CI:

- часто `workspace-write` + `never`, но только внутри надежно sandboxed runner.

Для production-adjacent tooling:

- read-only by default;
- write only through explicit reviewed playbooks.

### 17.2. Разделяйте доверие по задачам

Одни и те же настройки не подходят для:

- codebase exploration;
- docs generation;
- hotfix in incident;
- migration generator;
- infra change;
- release step.

### 17.3. Тестируйте safety contract

Очень полезный community-урок: после upgrade Codex CLI проверяйте:

- текущие feature flags;
- approval behavior;
- sandbox behavior;
- что делает `/status`;
- что реально считается trusted/untrusted в вашей среде.

### 17.4. Не полагайтесь только на prompt-level "не делай X"

Если действие реально опасно, одного текстового запрета недостаточно. Нужны:

- sandbox;
- approvals;
- rules;
- ограниченные writable roots;
- ограниченные tool surfaces.

## 18. Антипаттерны, которые ломают большие проекты в Codex CLI

Вот список ошибок, которые встречаются чаще всего.

### 18.1. Один огромный универсальный prompt вместо системной организации

Это почти всегда плохо. Prompt быстро устаревает, копируется, расходится между людьми и сессиями.

### 18.2. Параллельные сессии в одном и том же worktree

Почти всегда источник конфликтов, затертых изменений и плохого review.

### 18.3. Отсутствие явного планирования на крупных задачах

Когда агент сразу идет в код, большие задачи начинают расползаться.

### 18.4. Отсутствие ownership

Если неясно, кто отвечает за какой модуль/подзадачу/ветку, параллельная работа превращается в гонку.

### 18.5. Слепое доверие экспериментальным возможностям

Experimental surface хорош для controlled adoption, но опасен как безусловный фундамент критичного процесса.

### 18.6. Автоматизация через парсинг TUI

Если вам нужен API-like behavior, используйте `exec`, schema, server surfaces или SDK.

### 18.7. Смешивание hotfix и refactor

Типичная агентная ловушка: "раз уж я здесь, я еще и поправлю архитектуру". Нельзя.

### 18.8. Недокументированный handoff

Через сутки после длинной сессии без handoff даже вы сами потратите лишнее время на восстановление контекста.

## 19. Практический operating model для команды из нескольких инженеров

Ниже я даю модель, которая хорошо подходит большинству engineering teams.

### 19.1. Роли

- 1 человек или одна lead-сессия отвечает за decomposition и integration.
- Остальные Codex sessions/agents работают как bounded executors.
- Review всегда отдельным шагом.

### 19.2. Артефакты

Обязательные:

- `AGENTS.md`;
- project docs/architecture docs;
- issue tracker;
- `spec.md`/`plan.md`/`tasks.md` для крупных работ;
- handoff notes.

Желательные:

- Skills;
- MCP registry notes;
- compatibility matrix по Codex features;
- release checklist;
- incident playbooks.

### 19.3. Workflow

1. Issue или feature intake.
2. Исследование и clarification.
3. Spec/design.
4. Task decomposition.
5. Раскладка по worktrees.
6. Реализация.
7. Review.
8. Merge.
9. Документирование lessons learned.

### 19.4. Инструменты

Минимальный practical stack:

- git worktrees;
- Codex CLI profiles;
- `AGENTS.md`;
- Skills;
- MCP for docs/issues/logs;
- `codex review`;
- headless `codex exec` for automation.

Расширенный stack:

- session browser;
- desktop command center;
- tmux farm;
- headless SDK/adapters;
- app-server based internal tools.

## 20. Пример целевого setup для сложного проекта

Ниже не "единственно правильная" конфигурация, а практический reference architecture.

### 20.1. Структура процесса

- Глобальный `~/.codex/config.toml` с профилями.
- В репозитории есть `AGENTS.md`.
- В репозитории есть каталог со Skills.
- В репозитории есть docs для архитектуры и процессов.
- Подключены 2-4 MCP сервера по делу.
- Для больших задач есть spec artifacts.
- Для параллельной работы используются worktrees.

### 20.2. Запуск исследовательской сессии

```bash
codex -p research -C /path/to/repo
```

Задача:

- собрать архитектурный контекст;
- не писать код;
- понять границы изменений.

### 20.3. Запуск implementation session

```bash
codex -p implementation -C /path/to/worktree
```

Задача:

- делать кодовые изменения;
- держаться в пределах конкретного issue/stream;
- прогонять локальные проверки;
- готовить handoff.

### 20.4. Запуск review session

```bash
codex review --base main
```

Или:

```bash
codex review --uncommitted
```

### 20.5. Запуск CI automation

```bash
codex exec --json --output-last-message report.txt -p ci "Review changes against main and return a structured summary."
```

Если нужен строгий контракт:

```bash
codex exec --output-schema schema.json --output-last-message result.json -p ci "Analyze the diff and return findings."
```

## 21. Как организовать параллельную работу по направлениям

Разберем типичный большой проект с несколькими потоками:

- `feature/payments-v2`
- `fix/cache-regression`
- `infra/observability-cleanup`
- `docs/onboarding-rewrite`

Правильная схема:

### Поток 1. Общий planning stream

Одна lead-сессия:

- собирает контекст;
- пишет spec/plan;
- решает decomposition;
- выдает ownership.

### Поток 2. Реализация по отдельным worktrees

По одному worktree на направление.

### Поток 3. Внутренние bounded subagents

Внутри каждого worktree:

- explorer для локального code search;
- reviewer для self-check;
- test-writer/QA helper при необходимости.

### Поток 4. Headless quality gates

Независимые `codex review`/`codex exec` или CI jobs.

### Поток 5. Интеграция

Отдельная lead-сессия или человек:

- собирает результаты;
- делает merge/rebase;
- закрывает issues;
- нормализует знания.

Эта схема кажется тяжелой только в начале. На масштабе она сильно экономит время.

## 22. Какая форма параллельности лучше в каких случаях

### Multi-agent внутри одной сессии

Используйте, когда:

- нужно быстро распараллелить чтение;
- write scopes изолированы;
- результат нужен сразу;
- это sidecar work.

### Несколько worktrees

Используйте, когда:

- есть отдельные направления работы;
- ожидаются независимые diffs;
- нужна высокая надежность;
- есть риск конфликтов.

### Headless batch jobs

Используйте, когда:

- нужна автоматизация;
- нет смысла в интерактивном управлении;
- нужен machine-readable output;
- работа должна встраиваться в пайплайн.

### Desktop or control plane tools

Используйте, когда:

- у вас много одновременно живых сессий;
- нужен session overview;
- требуется richer UX;
- часть команды не любит pure TUI.

## 25. Почему Codex CLI особенно силен на сложных проектах при правильной организации

Если подвести инженерный итог, Codex CLI хорошо работает на сложных проектах не потому, что "умеет писать код". Это умеют многие инструменты. Его сила раскрывается, когда вы используете сразу несколько уровней:

- repo-aware instructions;
- bounded automation;
- tool integration;
- session continuity;
- role-based decomposition;
- runtime policies;
- structured operational memory.

Иначе говоря, сила Codex CLI — не в одной "суперспособности", а в том, что это многоуровневая агентная среда разработки, которую можно встроить в реальный engineering process.

## 27. Итоговые рекомендации

Если сформулировать рекомендации предельно прямо:

1. Не используйте Codex CLI на сложном проекте как просто "чат в терминале".
2. Обязательно заведите `AGENTS.md`.
3. Все крупные задачи проводите через планирование и декомпозицию.
4. Параллельную работу изолируйте worktrees, а не только сессиями.
5. Multi-agent используйте для bounded subtasks, а не как замену проектного менеджмента.
6. Повторяемые workflow выносите в Skills.
7. Внешние источники истины подключайте через MCP.
8. Для автоматизации используйте `codex exec`, schema и headless interfaces.
9. Review и quality gates не отдавайте на полный произвол одной агентной сессии.
10. После апдейтов перепроверяйте feature maturity и safety behavior.

Если вам нужна одна формула зрелой эксплуатации Codex CLI, то я бы записал ее так:

`Codex CLI + AGENTS.md + Skills + MCP + Profiles + Worktrees + Review Gates + Headless Automation`

## 30. Финальная позиция

Codex CLI уже достаточно зрелый, чтобы быть основой серьезной инженерной работы, но только если вы относитесь к нему как к платформе, а не как к чат-инструменту. Сложный проект требует не "больше промптов", а больше структуры:

- структурированных правил;
- структурированной декомпозиции;
- структурированной изоляции;
- структурированной автоматизации.

Если вы внедрите именно это, Codex CLI становится очень сильным инструментом для:

- крупных feature streams;
- рефакторинга;
- исследований;
- code review;
- operational automation;
- масштабируемой параллельной работы.

Если не внедрите, то даже самая сильная модель будет периодически упираться в организационный хаос вашего процесса.

И это, пожалуй, главный вывод всего исследования.

## 31. План внедрения Codex CLI в команду на 30, 60 и 90 дней

Чтобы эта инструкция не осталась теоретической, полезно закончить ее внедренческим планом. Большинство команд ломаются не на понимании возможностей Codex CLI, а на попытке внедрить слишком много сразу. Правильнее рассматривать внедрение как staged rollout.

### Первые 30 дней: stabilise the basics

На первом этапе цель не в том, чтобы получить максимальную автоматизацию. Цель в том, чтобы не создать новый слой хаоса.

Что нужно сделать:

1. Зафиксировать рабочую версию `codex-cli` и договориться, кто следит за upgrade policy.
2. Подготовить корневой `AGENTS.md`.
3. Подготовить минимум два профиля: `research` и `implementation`.
4. Определить default sandbox/approval policy для локальной работы.
5. Начать использовать `codex review` хотя бы на части изменений.
6. Ввести правило: каждая нетривиальная задача получает task brief.
7. Для одной-двух больших задач начать использовать отдельные worktrees.

Чего не делать на этом этапе:

- не вводить сразу десятки skills;
- не подключать все возможные MCP;
- не строить сразу внутреннюю платформу;
- не переводить весь CI на Codex;
- не разрешать безлимитный dangerous access как норму.

Ожидаемый результат первых 30 дней:

- команда понимает, где Codex действительно помогает;
- агент начинает вести себя более предсказуемо;
- появляется базовая дисциплина по инструкциям и worktree;
- снижается процент "агент пошел не туда".

### 60 дней: formalize repeatable workflows

На втором этапе задача уже не просто "пользоваться Codex", а превращать успешные паттерны в стандарт.

Что нужно сделать:

1. Выделить 3-5 наиболее повторяемых workflow и оформить их как Skills.
2. Добавить project-specific или module-specific `AGENTS.md`, если репозиторий большой.
3. Подключить 1-3 действительно окупающихся MCP.
4. Ввести профили `review` и `ci`.
5. Формализовать handoff template.
6. Начать вести spec/plan/task artifacts для крупных изменений.
7. Договориться о naming conventions для worktrees и веток.

На этом этапе обычно уже видно, какие классы задач лучше всего решаются Codex:

- диагностические исследования;
- подготовка изменений в знакомой подсистеме;
- code review;
- docs/update chores;
- structured reports.

Ожидаемый результат 60 дней:

- повторяемые задачи делаются быстрее;
- параллельная работа становится менее конфликтной;
- у команды появляется единый язык описания режимов работы;
- новые инженеры быстрее входят в процесс.

### 90 дней: platformize what is worth platformizing

Третий этап нужен не всем. Он нужен тем, у кого Codex перестал быть "личным инструментом отдельных инженеров" и стал общекомандным execution layer.

Что имеет смысл делать:

1. Вынести часть рутинных сценариев в `codex exec`.
2. Строить structured outputs через schema для downstream automation.
3. Подключить GitHub Action или другой CI integration для review/reporting workflows.
4. Ввести session browser, command center или tmux/zellij conventions как стандарт команды.
5. Добавить compatibility review для upgrades CLI и feature flags.
6. Описать incident, hotfix и release playbooks.
7. При необходимости испытать app-server, mcp-server или external orchestrator.

Ожидаемый результат 90 дней:

- Codex становится частью процесса разработки, а не "личным режимом работы одного человека";
- review и automation получают агентный слой без полной потери контроля;
- сложные задачи параллелятся через worktrees, а не через путаницу в одном дереве;
- команда понимает, где границы ответственности агента.

## 32. Практические шаблоны запуска для сложных сценариев

Ниже я даю не "идеальные промпты", а operational templates, которые хорошо работают именно на сложных проектах. Их ценность в структуре, а не в точной формулировке.

### 32.1. Шаблон на исследование большого модуля

Используйте, когда нужно быстро получить map незнакомой подсистемы.

```text
Изучи подсистему <name> в пределах <paths>.
Сначала собери контекст и не меняй файлы.
Нужно ответить:
1. Какова роль подсистемы.
2. Какие ключевые entry points и зависимости.
3. Какие риски изменения поведения.
4. Какие файлы критичны.
5. Что нужно проверить перед реализацией задачи <task>.
Если контекст неоднозначен, сначала собери доказательства из кода и конфигов.
```

Почему это работает:

- агент понимает, что исследование важнее раннего кодинга;
- сразу задается expected output;
- ограничивается scope;
- снижается риск преждевременных правок.

### 32.2. Шаблон на plan-first крупной задачи

```text
Перед изменением кода подготовь план реализации задачи <task>.
Нужно:
1. Уточнить затронутые модули.
2. Выявить риски и неизвестные.
3. Предложить пошаговый план.
4. Отдельно обозначить, что можно распараллелить.
5. Отдельно обозначить, где нужен отдельный worktree.
Файлы не редактировать, пока план не будет сформулирован.
```

Это особенно полезно на рефакторингах, миграциях и multi-module feature work.

### 32.3. Шаблон на bounded implementation

```text
Реализуй только изменения для <scope>.
Write scope ограничен путями <paths>.
Не меняй соседние подсистемы без явной необходимости.
Перед редактированием коротко опиши план.
После реализации:
1. Проверь затронутые тесты и сборку.
2. Укажи риски.
3. Подготовь handoff summary.
```

Смысл в том, чтобы не давать агенту психологическое разрешение "улучшать все вокруг".

### 32.4. Шаблон на post-implementation review

```text
Проведи code review изменений против <base>.
Сфокусируйся на:
1. behavioral regressions;
2. missing tests;
3. unsafe assumptions;
4. security and data-flow risks;
5. merge and rollout risks.
Если явных проблем нет, укажи residual risks и что еще стоило бы проверить.
```

### 32.5. Шаблон на multi-agent decomposition

```text
Разложи задачу <task> на подзадачи для параллельной работы.
Нужно:
1. Выделить независимые write scopes.
2. Отделить research subtasks от implementation subtasks.
3. Указать, что делать в отдельном worktree.
4. Указать, что можно делегировать subagent внутри текущей сессии.
5. Предложить порядок интеграции результатов.
```

Это один из самых полезных operational prompts, когда вы работаете не в одиночку.

## 33. Метрики зрелости и контроля качества для Codex CLI

Большинство команд измеряют только "ускорились или нет". Это слишком грубый показатель. Чтобы внедрение Codex CLI было управляемым, нужны более точные метрики.

### 33.1. Метрики предсказуемости

Смотрите на:

- сколько раз агент пошел в неверный write scope;
- сколько раз пришлось вручную останавливать рискованный курс;
- сколько задач пришлось "перезапускать с нуля" из-за плохой декомпозиции;
- сколько handoff notes оказались недостаточными.

Если эти метрики высокие, проблема обычно не в модели, а в слабых инструкциях и плохом task framing.

### 33.2. Метрики интеграции

Смотрите на:

- сколько параллельных worktrees удалось слить без болезненного конфликта;
- сколько PR после Codex-реализации требуют тяжелого ручного переписывания;
- сколько изменений проходят `codex review` и человеческое review без критических замечаний;
- сколько задач упираются в невнятный final diff.

Если тут проблемы, чаще всего не хватает ownership discipline и bounded scope.

### 33.3. Метрики повторяемости

Смотрите на:

- сколько процессов уже вынесено в Skills;
- сколько routine workflows идут через `codex exec`;
- сколько task categories имеют стандартный profile;
- насколько быстро новый инженер воспроизводит типовой workflow.

Повторяемость — главный признак того, что Codex перестал быть ad-hoc tool и стал частью platform workflow.

### 33.4. Метрики безопасности

Смотрите на:

- были ли неожиданные запросы на опасные команды;
- есть ли drift между ожидаемым и фактическим sandbox behavior;
- все ли dangerous операции проходят через понятную approval path;
- не накопилось ли слишком много broad MCP access.

### 33.5. Метрики качества контекста

Очень полезно раз в месяц проводить короткий audit:

- `AGENTS.md` не разросся ли до непригодного состояния;
- есть ли устаревшие instructions;
- какие Skills перестали соответствовать реальности;
- не появились ли новые крупные workflows, требующие formalization;
- достаточно ли кратки и полезны task briefs.

## 34. Как понять, что вы уже используете Codex CLI правильно

Есть несколько простых, но очень показательных симптомов зрелости.

### Признак 1. Вы меньше пишете длинные промпты вручную

Это хороший знак. Он означает, что контекст теперь живет не в голове и не в каждом чате заново, а в структуре проекта: `AGENTS.md`, skills, profiles, docs, MCP.

### Признак 2. Параллельная работа перестала быть страшной

Если у команды появились устойчивые паттерны:

- отдельный worktree на существенную задачу;
- один issue = один lifecycle;
- bounded write scopes;
- merge checkpoints;

значит вы реально освоили agent-assisted parallel development.

### Признак 3. У вас есть разные режимы работы, а не один "универсальный Codex"

Если у вас различаются:

- исследование;
- реализация;
- review;
- CI;
- hotfix;

значит вы мыслите системно.

### Признак 4. Вы можете объяснить, почему агенту разрешено именно это

Зрелая команда всегда может ответить:

- почему такой sandbox;
- почему такой approval policy;
- почему такой MCP set;
- почему эта задача идет через `exec`, а не через TUI;
- почему эта подзадача отдельным worktree.

Если ответа нет, значит процесс еще не стабилизирован.

### Признак 5. После апдейтов вы проверяете compatibility, а не надеетесь на удачу

Codex CLI активно развивается. Зрелое использование означает:

- читать changelog;
- проверять feature maturity;
- обновлять внутренние инструкции;
- тестировать чувствительные workflow после upgrade.

## 35. Окончательное резюме

Codex CLI на март 2026 года уже достаточно мощный, чтобы вести сложные проекты, но только в той мере, в какой вы готовы относиться к нему как к инженерной платформе. Его реальные сильные стороны проявляются не в "магии ответа", а в сочетании нескольких официально поддерживаемых слоев:

- интерактивные сессии;
- headless `exec`;
- `review`;
- `resume` и `fork`;
- `AGENTS.md`;
- Skills;
- MCP;
- config profiles;
- sandbox and approvals;
- evolving multi-agent workflows.

Практика сообщества усиливает этот вывод. Самые ценные реальные приемы вокруг Codex CLI почти всегда сводятся к одним и тем же вещам:

- spec-first мышление;
- worktree-first параллелизм;
- минимизация write conflict;
- формализация повторяемых workflow;
- отдельный integration layer;
- осторожное подключение внешних инструментов и экспериментальных поверхностей.

Если коротко, то на сложных проектах Codex CLI надо внедрять не как "еще одного умного разработчика", а как контролируемый execution runtime внутри вашей инженерной системы. Когда вы делаете именно так, агентная работа начинает масштабироваться.
