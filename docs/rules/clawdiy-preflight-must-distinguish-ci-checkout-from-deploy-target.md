# Clawdiy preflight должен различать CI checkout и deploy-target (RCA-014)

**Статус:** Active  
**Дата вступления:** 2026-03-14  
**Область действия:** `scripts/preflight-check.sh`, `deploy-clawdiy.yml`, статические тесты и все будущие проверки разворачивания Clawdiy

## Какую проблему предотвращает это правило

Путь `data/clawdiy/runtime` является live `runtime home` OpenClaw и обязан быть записываемым на реальном целевом сервере. Но в CI checkout этот каталог может еще не существовать, потому что он создается позже на шагах `render` и `deploy`. Если `preflight` не различает эти среды, он блокирует выкаты ложной ошибкой еще до фактического deploy.

## Обязательный протокол

1. Проверки в `--ci` режиме должны валидировать структуру и совместимость репозитория, а не требовать уже материализованный host runtime.
2. Проверки, зависящие от реального владельца путей и их materialization, должны быть жесткими только на deploy-target или после шага materialization.
3. Для Clawdiy отсутствие `data/clawdiy/runtime` в CI checkout не считается ошибкой, если deploy/render затем обязаны создать этот каталог.
4. Статические тесты должны отдельно фиксировать это разделение.

## Жесткое ограничение

Для Clawdiy запрещено:

- требовать существование `data/clawdiy/runtime` в `preflight --ci` как обязательного условия успеха
- требовать ownership `1000:1000` в локальном checkout GitHub runner
- смешивать локальный repository validation и remote target validation в одном безусловном check

## Ожидаемое поведение

- CI проверяет, что deploy/render знают, как создать `runtime home`.
- Реальный deploy-target проверяет, что `runtime home` существует и принадлежит runtime-пользователю.
- Workflow не падает преждевременно на локальном раннере, но сохраняет fail-closed поведение на реальном сервере.
